## Sub-session s1: Norton TLS Fix + Manual SAS Rotation

### Context

Entered this session with system in DEGRADED state since 2026-05-01 (SAS expired, backups failing silently). The rotation itself had been blocked by an SSL certificate verification failure in Azure CLI. Root cause: Norton Antivirus "Web/Mail Shield" SSL scanning feature intercepts all HTTPS traffic and re-signs it with a self-generated root CA (`CN=Norton Web/Mail Shield Root`). Windows trusted this cert (Norton pushed it to the Windows Certificate Store). Azure CLI's bundled Python uses its own `certifi` CA bundle — which does not know Norton's root.

### Resolution

Added Azure endpoint domains to Norton's SSL scanning exclusion list one family at a time:
1. `login.microsoftonline.com`
2. `*.vault.azure.net`
3. `*.blob.core.windows.net`
4. `management.azure.com`

After exclusions, manual SAS rotation completed:
- New token generated: 2026-05-08T00:35:42Z
- New expiry: 2026-05-14T20:35:40Z (7-day Azure maximum for user-delegation SAS)
- Token length: 269 chars; BOM-free confirmed (first char code 115, not 65279)
- KV write confirmed: 2026-05-08T00:35:57Z
- **System state: LIVE ✅**

### Key Technical Learnings (s1)

- Two trust stores co-exist on one machine: Windows Certificate Store (used by .NET/PowerShell) and Python's `certifi` bundle (used by Azure CLI). A cert trusted by Windows may be unknown to Python on the same host.
- User-delegation SAS max is 7 days — the Azure hard cap. This constraint makes manual weekly rotation unsustainable → motivates component 08.
- **Never use `--value` with SAS tokens** — the `&` characters break CLI argument parsing. Always use `--file` with a BOM-free UTF-8 temp file.
- **Never use `Out-File -Encoding utf8`** — PowerShell 5.1 adds a 3-byte BOM. Use `System.Text.UTF8Encoding($false)`.

## Sub-session s2: Component 08 — SAS Rotation Automation (Azure Automation Account)

### Context

With system LIVE again, the permanent fix for the SAS expiry problem was built. Component 08 was originally planned as an Azure Function (Consumption plan). During deploy, the personal/PAYG Azure subscription returned Dynamic VM quota = 0. The Function App provisioned but the runtime container could not start.

**Pivot:** Azure Automation Account (Free SKU) — same system-assigned MSI model, no Dynamic VM quota requirement, $0/month at 1 invocation per 6 days.

### Architectural Decision (ADR-008)

See `docs/decisions/ADR-008-sas-rotation-automation.md` for the full record. Key points:

- **Chosen:** Azure Automation Account with MSI, PowerShell runbook, 6-day schedule
- **Rejected:** Azure Functions (quota wall), workstation scheduled task (no MSI on non-Azure compute), account-key SAS (violates `allowSharedKeyAccess: false` constraint), KV rotation policy (no native SAS support)
- **RBAC:** MSI granted `Key Vault Secrets Officer` scoped to the specific secret resource ID (not the vault); `Storage Blob Delegator` + `Storage Blob Data Contributor` on the container (minimum required for user-delegation SAS with write permissions)
- **Overlap:** Generates 6d 23h token on a 6-day schedule — 23-hour window ensures old token remains valid even if a run fires late

### az CLI Gaps Discovered

Two critical Automation operations are missing from the `az automation` CLI extension:
- `az automation runbook replace-content` — does not exist; must use REST API PUT with `Content-Type: text/powershell`
- `az automation jobSchedules create` — does not exist; must use REST API PUT on the `jobSchedules` resource with a Bearer token

`Deploy.ps1` handles both via `Invoke-RestMethod`. This is documented in the CLAUDE.md architecture section and in ADR-008.

- Job ID: `45bb7628`
- Status: Completed

### Cross-Agent Review: Automation MSI Trust Zone

🛡️ **Security Engineer:** The Automation Account MSI is the widest credential in the system — it can write directly to `lab-files` without a SAS intermediary. Mitigations: (1) MSI token is ephemeral and cannot be exported; (2) immutability policy prevents delete even with direct write; (3) every job invocation logged to Log Analytics. Named as trust zone Z9 in `threat-model.md §2`.

🏗️ **Architect:** The runbook upload via REST is inelegant but the right call — documenting the CLI gap explicitly is better than pretending Bicep alone handles it. The 6-day/23h overlap is robust against single-run failures.

🔧 **Operator:** Failure mode is safe — old token stays valid during the overlap window. `Write-Error` + `throw` in the runbook marks the job Failed in Automation history, which surfaces in Log Analytics. Monthly health checks documented in RUNBOOK.md.

📚 **Documentarian:** Component README, ADR-008, and RUNBOOK rotation procedures all consistent as of commit `8f0bf26`.
