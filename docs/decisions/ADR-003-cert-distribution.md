# ADR-003: Service Principal cert generation and distribution

**Status:** Accepted
**Date:** 2026-04-30
**Agents who reviewed:** 🛡️ 🏗️ 🔧

## Decision

The Service Principal cert is **generated by `Deploy.ps1` at deploy time** using `az ad sp create-for-rbac --create-cert --years 0.25`. The cert PEM is emitted to `./out/<prefix>-sp-cert-<timestamp>.pem`. The operator distributes it to lab workstations through their chosen channel.

**SP creation itself is imperative**, not Bicep. SP/Application creation is a Microsoft Graph operation, not an ARM operation. Bicep has limited (preview) support via `Microsoft.Graph` types or via `Microsoft.Resources/deploymentScripts` shims. Both add complexity without changing the security posture.

## Alternatives considered

1. **Pure Bicep with `Microsoft.Graph/applications` (preview)** — would let the entire deploy be declarative. Rejected because Graph Bicep types are still preview-quality (early 2026), and we don't want a turnkey production design depending on preview features.

2. **`Microsoft.Resources/deploymentScripts`** — runs an ACI container with az CLI to create the SP. Rejected because: (a) it's a moving part with its own RBAC requirements, (b) the container itself costs money and runtime, (c) the cert handling inside the container is harder to make secure than in the operator's local PowerShell session.

3. **External CA-signed cert (operator pre-generates)** — operator generates a cert from internal PKI, hands the public key (.cer) to deploy via a parameter, deploy attaches it to the SP. **Recommended for hardened deploys.** Captured below as the second mode.

## Two modes

### Mode A: Self-signed cert generated by deploy (default)
- `az ad sp create-for-rbac --create-cert` produces the keypair on the deploy host
- The private key (PEM) lives in `./out/` until the operator distributes and deletes
- 90-day cert lifetime
- Pro: zero PKI dependency, works for first-time deployers
- Con: the deploy host briefly holds the private key — defense is "delete after distribution"

### Mode B: External CA-signed cert (hardened)
- Operator runs their PKI to issue a cert (any algorithm Entra ID accepts)
- Operator passes only the public key (`-CertPublicKey ./mycert.cer`) to deploy
- Private key never touches the deploy host
- Pro: strongest distribution model
- Con: requires PKI infrastructure; not accessible to small operators

`Deploy.ps1` currently implements Mode A. Mode B is a documented hardening path for v2.

## Distribution channel

Documented options (operator picks one):

1. **In-person USB.** Highest-trust; bandwidth-limited to walking distance.
2. **Encrypted email with PFX password sent through different channel.** Common, defensible.
3. **Internal PKI cert distribution (Mode B).** Best for orgs with existing PKI.
4. **GPO software distribution (signed MSI).** For centrally-managed lab PC fleets.

Whatever channel is used, the operator MUST:
- Delete the .pem from the deploy host within 24 hours of distribution completion
- Wipe the deploy host's recycle bin
- Confirm distribution with at least a thumbprint check on each lab PC

## Trade-off

We accept that Mode A holds the private key on the deploy host briefly. This is the price of a turnkey deploy that does not require pre-existing PKI infrastructure. The threat model T2 (lifted credential) defenses (write-only RBAC, immutable destination, time-bounded cert) bound the blast radius even if Mode A's distribution is sloppy.

## Cross-Agent Review

- 🛡️ Security Engineer: Signed with reservation. Mode A is the lesser evil for a turnkey design; document the upgrade path to Mode B clearly. The post-distribution deletion procedure must be in RUNBOOK.md.
- 🏗️ Architect: Signed.
- 🔧 Operator: Signed. The two-mode structure is right.
